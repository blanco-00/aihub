## 数据库表设计

> 📝 **说明**：完整的SQL表结构定义请参考后端迁移脚本 `backend/aihub-api/src/main/resources/db/migration/V1.0.0__init_tables.sql`

### 登录日志表 (login_log)

**主要字段**：
- `id`: 日志ID（主键）
- `user_id`: 用户ID
- `username`: 用户名
- `ip`: IP地址
- `address`: 登录地址
- `user_agent`: 用户代理（浏览器信息）
- `status`: 登录状态（0-失败，1-成功）
- `message`: 登录消息
- `login_time`: 登录时间

### 操作日志表 (operation_log)

**主要字段**：
- `id`: 日志ID（主键）
- `user_id`: 用户ID
- `username`: 用户名
- `module`: 操作模块
- `operation`: 操作类型
- `method`: 请求方法（GET/POST/PUT/DELETE等）
- `url`: 请求URL
- `params`: 请求参数
- `result`: 操作结果
- `status`: 操作状态（0-失败，1-成功）
- `ip`: IP地址
- `duration`: 耗时（毫秒）
- `operation_time`: 操作时间

### 系统日志表 (system_log)

**主要字段**：
- `id`: 日志ID（主键）
- `level`: 日志级别（DEBUG/INFO/WARN/ERROR）
- `module`: 模块名称
- `message`: 日志消息
- `stack_trace`: 堆栈信息
- `ip`: IP地址
- `user_id`: 用户ID
- `request_id`: 请求ID
- `log_time`: 日志时间

## 在线用户设计

### 存储方式

使用 Redis 存储在线用户信息：

- **Key**: `online:user:{userId}`
- **Value**: JSON（包含登录时间、IP、Token等信息）
- **TTL**: Token过期时间

### 功能设计

- **查询接口**：从 Redis 读取所有在线用户
- **强制下线**：删除 Redis 中的 Token，前端刷新时自动跳转登录
- **在线人数统计**：统计当前在线用户数量

## 运行状况监控

### 功能说明

运行状况监控提供实时的系统监控数据，包括：
- **主机信息**：CPU、内存、磁盘使用率，系统运行时间等
- **JVM信息**：堆内存、非堆内存、线程数、GC信息等
- **MySQL信息**：连接池状态、版本、连接数等
- **Redis信息**：内存使用、连接数、键数量等

### API接口

- **GET** `/api/monitor/system` - 获取系统监控信息

### 性能优化设计

> ⚠️ **重要**：监控功能已进行性能优化，确保不影响系统性能。以下为详细的优化策略。

#### 1. 数据缓存机制

**问题**：监控数据获取涉及系统调用、数据库查询、Redis查询等操作，频繁请求会消耗大量系统资源。

**解决方案**：
- 使用内存缓存，缓存监控数据 **3秒**
- 在缓存有效期内，直接返回缓存数据，避免重复计算
- 缓存失效后，才重新获取最新数据

**实现位置**：
- `SystemMonitorServiceImpl` 使用 `AtomicReference` 和 `AtomicLong` 实现线程安全的缓存

#### 2. CPU使用率计算优化

**问题**：CPU使用率计算需要等待1秒才能获取准确的数值，如果每次请求都sleep 1秒，会导致接口响应很慢。

**解决方案**：
- 采用增量计算方式：第一次获取时记录CPU ticks，下次请求时计算差值
- 如果距离上次更新不足1秒，使用缓存中的CPU使用率
- 避免每次请求都sleep，提升响应速度

**实现位置**：
- `SystemMonitorServiceImpl.getHostInfo()` 方法中的CPU使用率计算逻辑

#### 3. 静态信息缓存

**问题**：系统版本、主机名、IP地址、JVM版本等静态信息不会变化，每次请求都重新获取浪费资源。

**解决方案**：
- 静态信息只获取一次，使用 `volatile` 变量缓存
- 后续请求直接使用缓存值，不再重复获取

**缓存的信息**：
- 操作系统名称、版本、架构
- 主机名、IP地址
- CPU核心数
- JVM名称、版本、Java版本

**实现位置**：
- `SystemMonitorServiceImpl` 中的静态信息缓存变量

#### 4. 数据库连接优化

**问题**：每次获取数据库信息都创建新连接，影响性能。

**解决方案**：
- 使用连接池（HikariCP）复用连接
- 直接从连接池获取统计信息，无需额外查询
- 使用 try-with-resources 确保连接正确归还

**实现位置**：
- `SystemMonitorServiceImpl.getDatabaseInfo()` 方法

#### 5. Redis连接优化

**问题**：每次获取Redis信息都创建新连接，影响性能。

**解决方案**：
- 使用Redis连接池复用连接
- 复用 `RedisTemplate` 的连接，避免频繁创建
- 使用 try-finally 确保连接正确归还连接池

**实现位置**：
- `SystemMonitorServiceImpl.getCacheInfo()` 方法

#### 6. 前端请求频率限制

**问题**：前端频繁请求监控接口，即使有缓存也会增加不必要的网络开销。

**解决方案**：
- 前端自动刷新间隔设置为 **5秒**（大于后端缓存时间3秒）
- 提供手动刷新按钮，用户可按需刷新
- 支持关闭自动刷新，减少不必要的请求

**实现位置**：
- `frontend/src/views/monitor/system/utils/hook.tsx` 中的 `startAutoRefresh` 方法
- 前端页面提供自动刷新开关

### 性能指标

#### 缓存策略

| 数据类型 | 缓存时间 | 说明 |
|---------|---------|------|
| 监控数据 | 3秒 | 整体监控数据缓存时间 |
| CPU使用率 | 1秒 | CPU更新间隔 |
| 静态信息 | 永久 | 系统版本、主机名等静态信息 |

#### 响应时间

- **缓存命中**：< 10ms（直接返回缓存数据）
- **缓存未命中**：50-200ms（重新计算，取决于系统负载）

#### 资源消耗

- **内存占用**：< 10KB（缓存一个监控响应对象）
- **CPU占用**：< 1%（缓存命中时几乎无CPU消耗）
- **数据库连接**：复用连接池，无额外连接开销
- **Redis连接**：复用连接池，无额外连接开销

### 使用建议

#### 后端

1. **不要频繁调用监控接口**：即使有缓存，频繁请求也会增加网络开销
2. **监控数据缓存3秒**：如需更实时的数据，可考虑使用WebSocket推送
3. **静态信息已缓存**：系统版本等信息不会变化，无需频繁获取

#### 前端

1. **自动刷新间隔**：建议设置为5秒或更长，避免不必要的请求
2. **手动刷新**：提供手动刷新按钮，用户可按需获取最新数据
3. **关闭自动刷新**：长时间不查看监控时，建议关闭自动刷新

### 未来优化方向

1. **WebSocket推送**：对于需要实时监控的场景，可考虑使用WebSocket推送监控数据
2. **异步更新**：使用定时任务异步更新监控数据，接口直接返回缓存
3. **数据采样**：对于历史监控数据，可进行采样存储，减少存储空间
4. **监控告警**：当监控指标超过阈值时，主动推送告警，而非被动查询

### 注意事项

1. **缓存时间**：当前缓存时间为3秒，如需调整，修改 `CACHE_DURATION_MS` 常量
2. **CPU计算**：首次请求CPU使用率可能为0，需要等待1秒后再次请求才能获取准确值
3. **连接池配置**：确保数据库和Redis连接池配置合理，避免连接数不足
4. **异常处理**：监控数据获取失败时，返回缓存数据（如果有），避免接口报错
